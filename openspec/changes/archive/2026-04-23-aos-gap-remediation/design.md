## Context

The AOS (Architecture Operating System) is a Git-native, Markdown + YAML framework for capturing, linking, and evolving architectural decisions. A structured review identified seven gaps that prevent reliable operation at scale. This design covers how to implement the remediation across three phases, using only file-based tooling consistent with the AOS philosophy.

Current state: AOS has a working folder structure (`/architecture/inbox/`, `/architecture/adr/`, `/architecture/model/`) but no schema enforcement, no lifecycle states, no ownership definition, and no automated graph generation.

Stakeholders: Architecture lead, domain contributors, CI pipeline.

## Goals / Non-Goals

**Goals:**
- Define YAML schemas for all AOS file types and enforce them in CI
- Add ADR status lifecycle with a formal state machine and CI validation
- Name accountable owners for every artifact type via `OWNERS.yaml` and `CODEOWNERS`
- Auto-generate a queryable knowledge graph from existing YAML and ADR files
- Add staleness detection for service model entries
- Reduce input capture friction with draft status and a CLI helper
- Define a PR-label-based RFC process for multi-architect governance

**Non-Goals:**
- Building a graph database or dedicated query engine — file-based `yq` queries are sufficient
- Replacing Git PR review workflow with a custom tool
- Real-time Slack/Jira integration (Phase 3 only, design deferred)
- Migrating existing ADRs automatically — backfill is a manual one-time task

## Decisions

### Decision 1: YAML validation via `yamale` over JSON Schema

**Choice:** `yamale` (Python) with `.schema.yaml` files co-located in a `schemas/` directory.

**Rationale:** JSON Schema requires a separate validation runner and produces verbose, hard-to-read schema files. `yamale` uses a YAML-native syntax that mirrors the files being validated, keeping the tool consistent with the AOS all-YAML philosophy. It integrates directly into a `pip install yamale && yamale -s schema file` CI one-liner with no additional config.

**Alternatives considered:**
- `pykwalify` — more powerful but requires separate YAML config and is less readable
- JSON Schema + `check-jsonschema` — correct choice for OpenAPI schemas, but unnecessary ceremony here

---

### Decision 2: Knowledge graph as a flat YAML index, not a graph database

**Choice:** `architecture/index.yaml` generated by `scripts/aos-index.py`, queried with `yq`.

**Rationale:** The graph is already implicit in the files — problems reference services, ADRs reference problems and services. The generator script simply materialises these links into a single flat index. No database is needed because the query volume is low (architect-pace, not application-pace), and `yq` is already a standard infrastructure tool.

**Alternatives considered:**
- SQLite database — adds a binary file to Git, complicates diffs, requires a query interface
- Neo4j / ArangoDB — massively over-engineered for this scale; violates the AOS anti-pattern of tool-heavy architecture

---

### Decision 3: ADR lifecycle enforced at CI merge time, not write time

**Choice:** A CI script validates that all ADR files contain a `status` field conforming to the state machine before merge. No editor plugin or pre-commit hook required.

**Rationale:** Pre-commit hooks are skippable (`git commit --no-verify`) and require contributor toolchain setup. CI enforcement is authoritative and requires no tooling beyond what already exists. The check is a simple frontmatter parse — trivially fast.

**Alternatives considered:**
- ADR template tooling (e.g., `adr-tools`) — locks the format to an external tool; incompatible with the AOS custom frontmatter
- Pre-commit hook only — too easy to bypass; not suitable as the primary enforcement mechanism

---

### Decision 4: Ownership via `OWNERS.yaml` + `CODEOWNERS`, not a custom system

**Choice:** `OWNERS.yaml` at the root defines roles and policies in human-readable form. `.github/CODEOWNERS` enforces PR review requirements via native GitHub/GitLab functionality.

**Rationale:** `CODEOWNERS` is natively supported by all major Git platforms and requires zero tooling. It integrates with existing PR workflows. `OWNERS.yaml` is the human-readable contract — it declares intent, SLAs, and escalation paths that `CODEOWNERS` cannot express.

---

### Decision 5: Staleness as a CI warning, not a blocking error

**Choice:** `scripts/check-staleness.py` exits `0` (warning only) in the default configuration. Teams can opt into blocking (`sys.exit(1)`) once the `last_reviewed` fields are fully backfilled.

**Rationale:** Immediately blocking CI on staleness would break existing pipelines since no entries have `last_reviewed` yet. A warning-first approach allows teams to adopt incrementally without a big-bang migration.

## Risks / Trade-offs

- **Backfill burden** — Existing ADRs need `status` fields and existing services need `last_reviewed` + `owner` fields. This is a one-time manual task. → Mitigation: provide a migration script template in `scripts/` and document it in `OWNERS.yaml`.
- **`yamale` Python dependency** — Adds a Python dependency to CI. → Mitigation: pin the version in `requirements-ci.txt`; the library is stable and actively maintained.
- **Knowledge graph drift** — If `aos-index.py` is not run in CI, the graph goes stale. → Mitigation: make index generation a required CI step, committing the updated `index.yaml` on push.
- **Draft inbox entries** — Allowing `status: draft` reduces capture friction but could create a large backlog of unresolved drafts. → Mitigation: enforce the 48-hour triage SLA in `OWNERS.yaml` and surface outstanding drafts in the weekly `#arch-health` report.
- **Single architecture lead bottleneck** — requiring the lead's approval on ADR acceptance could slow throughput on active teams. → Mitigation: `OWNERS.yaml` allows designating a deputy; revisit if triage queue exceeds 5 open ADRs.

## Migration Plan

**Phase 1 (immediate — no tooling required):**
1. Add `schemas/` directory with `problem.schema.yaml`, `service.schema.yaml`, `adr.schema.yaml`
2. Add CI step for YAML schema validation
3. Update ADR template to include `status`, `date`, `superseded_by` in `## Metadata`
4. Backfill `status` on all existing ADRs
5. Create `OWNERS.yaml` at repository root
6. Create or update `.github/CODEOWNERS`

**Phase 2 (short term):**
7. Write `scripts/aos-index.py` and add to CI
8. Add `last_reviewed` and `owner` to all service entries in `services.yaml`
9. Write `scripts/check-staleness.py` and add as CI warning step
10. Commit initial `architecture/index.yaml`

**Phase 3 (team growth):**
11. Define PR label lifecycle in repository settings
12. Document conflict resolution policy in `OWNERS.yaml`
13. Implement `aos capture` CLI stub (shell alias or Python script)

**Rollback:** All changes are additive file additions and CI config updates. Reverting any step is a standard Git revert with no data loss.

## Open Questions

- Should `aos-index.py` commit the updated `index.yaml` directly in CI (via a commit bot), or should it only validate that the checked-in `index.yaml` is up to date? (Recommendation: validate-only first, commit-bot in Phase 2 if validation becomes noisy)
- Who is the designated deputy for the architecture lead role when they are unavailable?
- Should `adr.schema.yaml` validate the Markdown body structure, or only the frontmatter `## Metadata` block? (Recommendation: frontmatter only — body validation is fragile via YAML schema)
